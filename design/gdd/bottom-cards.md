# C4 抠底/配底

> **Status**: Designed
> **Author**: user + game-designer
> **Last Updated**: 2026-04-06
> **Implements Pillar**: 支柱 2（规则即策略）

## Overview

**抠底/配底系统**负责庄家确定后的底牌处理流程。庄家从 F2 获得底牌，将底牌加入手牌后选择等量的牌扣下作为新底牌。扣下的底牌在对局结束时参与结算（C6 抠底倍数计算）。本系统也处理反主成功后的重新配底流程。

## Player Fantasy

配底是庄家的第一个重大策略决策——藏什么牌进底？分牌放底保安全？还是把副牌清掉强化主牌控制力？好的配底能奠定整局胜势，坏的配底会让防守方轻松抠底翻盘。本系统服务的情感是**策略掌控感**。

## Detailed Design

### Core Rules

#### 1. 抠底（查看底牌）

```
reveal_bottom(bottom: Card[], dealer_hand: Card[]) → Card[]
```

- 庄家获得底牌（F2 提供的 `bottom_size` 张牌）
- 底牌加入庄家手牌，此时庄家手牌 = `hand_size + bottom_size`
  - 2 副牌：25 + 8 = 33 张
  - 1 副牌：12 + 6 = 18 张

#### 2. 配底（选牌扣下）

```
bury_bottom(selected: Card[], hand: Card[], bottom_size: int) → Result<BuryResult, Error>
```

- 庄家从手牌中选择 `bottom_size` 张牌扣下
- 校验：选择张数 == `bottom_size`
- 校验：所有选中牌在当前手牌中存在
- 无其他限制：任何牌都可以扣底（包括分牌、主牌、级牌、王牌）
- 扣底完成后，庄家手牌恢复为 `hand_size` 张

#### 3. 反主后重新配底

反主成功时（C3 通知）：
1. 原庄家的底牌回收
2. 底牌交给反家
3. 反家执行完整的抠底+配底流程（同上）
4. 完成后通知 C3/C7

#### 4. 底牌的后续用途

扣下的底牌在对局中不可见（暗牌），对局结束时：
- 如果攻方赢得最后一墩 → 底牌中的分牌按抠底倍数计入攻方得分（C6 处理）
- 如果庄家方赢得最后一墩 → 底牌分值不计入任何一方

### States and Transitions

| 状态 | 说明 |
|------|------|
| **Idle** | 等待庄家确定 |
| **Revealed** | 底牌已展示给庄家，等待配底 |
| **Buried** | 配底完成 |
| **Reburying** | 反主成功，反家重新配底中 |
| **Done** | 最终配底完成，C4 职责结束 |

转换：
```
正常流程：  Idle → Revealed → Buried → Done
反主流程：  Buried → Reburying → Revealed → Buried → Done
```

### Interactions with Other Systems

| 方向 | 系统 | 接口 |
|------|------|------|
| ← F2 牌组管理 | 接收底牌 Card[] |
| ← C3 亮主/抢主/反主 | 接收庄家确定通知、反主成功通知 |
| ← C1 主副牌判定 | 庄家查看底牌时可参考主副判定排列手牌 |
| → C6 升级结算 | 底牌 Card[] 在对局结束时交给 C6 计算抠底分 |
| → C7 对局状态机 | 通知配底完成，进入出牌阶段 |

## Formulas

底牌分值计算归 C6 负责。C4 只负责底牌的选择与存储，无独立公式。

庄家配底期间手牌张数：`hand_size + bottom_size`

| deck_count | 配底期间手牌 | 配底后手牌 | 底牌 |
|------------|------------|-----------|------|
| 1 | 18 | 12 | 6 |
| 2 | 33 | 25 | 8 |

## Edge Cases

| # | 边界情况 | 处理方式 |
|---|---------|---------|
| E1 | **庄家把所有分牌扣底** | 合法——风险极高但规则允许 |
| E2 | **庄家把王牌扣底** | 合法——策略性选择（罕见但允许） |
| E3 | **反主后原底牌包含分牌** | 原底牌全部回收（包括分牌），反家重新配底 |
| E4 | **配底超时（未来联机场景）** | MVP 单机无超时。联机时需要定义默认行为（随机选牌？） |

## Dependencies

| 依赖方向 | 系统 | 类型 | 接口描述 |
|---------|------|------|---------|

| ← C3 亮主/抢主/反主 | 硬依赖 | 庄家确定 + 反主通知 |
| ← C1 主副牌判定 | 软依赖 | 手牌排序辅助 |

## Tuning Knobs

C4 无独立可调参数。底牌张数由 F3 的 `bottom_size`（从 `deck_count` 派生）决定。

## Acceptance Criteria

| # | 测试条件 | 预期结果 |
|---|---------|---------|
| AC1 | 庄家获得底牌（2副牌，8张） | 手牌变为 33 张 |
| AC2 | 庄家选 8 张配底 | 手牌恢复为 25 张，底牌 8 张 |
| AC3 | 庄家选 7 张配底（张数不足） | 拒绝 |
| AC4 | 庄家选了不在手牌中的牌 | 拒绝 |
| AC5 | 庄家把 5、10、K（分牌）全部扣底 | 合法 |
| AC6 | 反主成功 | 原底牌回收，反家手牌变为 33 张 |
| AC7 | 反家完成配底后 | 先手仍归原庄家，进入出牌阶段 |
